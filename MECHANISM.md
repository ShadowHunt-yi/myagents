# MyAgents 运行机制详解

## 一、完成后的全貌

完成后，你拥有一个**可插拔的 Agent 框架**。
用户说一句话，Agent 自动决定"要不要用工具"、"用哪个工具"、"用完之后怎么回答"。

```
你: 帮我算一下 (3+5)*12 再搜一下今天上海天气

Agent 内部自动完成:
  → 识别出需要两个工具
  → 调用 calculator 算出 96
  → 调用 web_search 搜到 "上海 28°C 多云"
  → 组织语言回答: "(3+5)×12=96。今天上海28°C，多云。"
```

你不需要写任何解析逻辑，**LLM 原生的 function calling 机制**替你完成了工具选择。

## 二、一次完整请求的生命周期

以用户问 **"23乘以47等于多少"** 为例，走一遍完整流程:

```
用户输入
  │
  ▼
┌──────────────────────────────────────────────────────────────────┐
│ 第1阶段: 组装上下文                                              │
│                                                                  │
│  Agent.build_context() 把以下内容拼成一个 messages 列表:          │
│                                                                  │
│  ┌─────────────────────────────────────────────┐                 │
│  │ messages[0] = system prompt                  │  "你是一个助手" │
│  │ messages[1] = 核心记忆（如果有）              │                │
│  │ messages[2] = 长期记忆（如果有）              │                │
│  │ messages[3] = 短期记忆（如果有）              │                │
│  │ messages[4...] = 历史对话                     │                │
│  │ messages[-1] = 用户当前输入                   │  "23乘以47"   │
│  └─────────────────────────────────────────────┘                 │
└──────────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────────┐
│ 第2阶段: 调用 LLM（带工具描述）                                   │
│                                                                  │
│  Agent 调用:                                                     │
│    self.llm.chat(                                                │
│        messages = [上面拼好的列表],                                │
│        tools = [                                                 │
│          {                                                       │
│            "type": "function",                                   │
│            "function": {                                         │
│              "name": "calculator",                               │
│              "description": "数学计算工具...",                    │
│              "parameters": {                                     │
│                "type": "object",                                 │
│                "properties": {                                   │
│                  "expression": {                                 │
│                    "type": "string",                             │
│                    "description": "数学表达式"                   │
│                  }                                               │
│                },                                                │
│                "required": ["expression"]                        │
│              }                                                   │
│            }                                                     │
│          },                                                      │
│          { ... web_search schema ... }                            │
│        ]                                                         │
│    )                                                             │
│                                                                  │
│  注意: tools 参数就是 ToolRegistry.get_all_schemas() 的输出       │
│  LLM 看到这些 schema，就知道有哪些工具可用、每个工具要什么参数     │
└──────────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────────┐
│ 第3阶段: LLM 决策                                                │
│                                                                  │
│  LLM 不是返回文字，而是返回一个结构化的 tool_call:                │
│                                                                  │
│  response = {                                                    │
│    "role": "assistant",                                          │
│    "content": null,              ← 注意: 没有文字内容！           │
│    "tool_calls": [                                               │
│      {                                                           │
│        "id": "call_abc123",      ← 唯一标识，后面要用            │
│        "type": "function",                                       │
│        "function": {                                             │
│          "name": "calculator",   ← LLM 选了计算器               │
│          "arguments": "{\"expression\": \"23*47\"}"              │
│        }                         ← LLM 自动构造了参数            │
│      }                                                           │
│    ]                                                             │
│  }                                                               │
│                                                                  │
│  这就是 function calling 的核心:                                  │
│  LLM 不是输出 "Action: calculator" 这种文本让你正则解析           │
│  而是直接返回结构化的 JSON，告诉你调哪个函数、传什么参数           │
└──────────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────────┐
│ 第4阶段: 执行工具                                                │
│                                                                  │
│  Agent 拿到 tool_call，调用:                                     │
│    result = self.tools.execute("calculator", {"expression":"23*47"})│
│                                                                  │
│  内部链路:                                                       │
│    ToolRegistry.execute()                                        │
│      → 查找 self._tools["calculator"]                            │
│      → 找到 CalculatorTool 实例                                  │
│      → 调用 CalculatorTool.run({"expression": "23*47"})          │
│      → AST 解析 → 计算 → 返回 "1081"                            │
└──────────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────────┐
│ 第5阶段: 把工具结果喂回 LLM                                      │
│                                                                  │
│  Agent 把两条新消息追加到 messages:                                │
│                                                                  │
│  messages.append({                                               │
│    "role": "assistant",                                          │
│    "tool_calls": [{"id":"call_abc123", ...}]                     │
│  })                              ← 告诉 LLM "你刚才选了工具"     │
│                                                                  │
│  messages.append({                                               │
│    "role": "tool",                                               │
│    "tool_call_id": "call_abc123",← 和上面的 id 对应              │
│    "name": "calculator",                                         │
│    "content": "1081"             ← 工具执行结果                  │
│  })                              ← 告诉 LLM "工具返回了这个"     │
│                                                                  │
│  然后再次调用 LLM:                                                │
│    self.llm.chat(messages)       ← 这次不传 tools，让 LLM 总结   │
└──────────────────────────────────────────────────────────────────┘
  │
  ▼
┌──────────────────────────────────────────────────────────────────┐
│ 第6阶段: LLM 生成最终回答                                        │
│                                                                  │
│  LLM 这次看到了工具结果 "1081"，直接输出自然语言:                 │
│                                                                  │
│  response = {                                                    │
│    "role": "assistant",                                          │
│    "content": "23 × 47 = 1081"   ← 最终回答，返回给用户         │
│  }                                                               │
└──────────────────────────────────────────────────────────────────┘
  │
  ▼
用户看到: "23 × 47 = 1081"
```

## 三、三种 Agent 策略的运行机制对比

三个 Agent 继承同一个基类，但 `run()` 内部的"思考策略"完全不同:

### 3.1 ReActAgent — 循环式工具调用

```
适用场景: 需要调用工具获取外部信息的任务
比如: "搜索今天的新闻" "帮我算个数" "查天气"

运行流程:
                    ┌──────────────┐
                    │   用户输入    │
                    └──────┬───────┘
                           ▼
                 ┌─── 调 LLM（带工具）───┐
                 │                       │
           有 tool_calls            无 tool_calls
                 │                       │
                 ▼                       ▼
           执行工具，拿到结果        直接返回文字
                 │                  （最终回答）
                 ▼
         把结果喂回 LLM
                 │
                 ▼
         ┌─ LLM 再次决策 ─┐
         │                 │
    还需要工具          不需要了
         │                 │
         ▼                 ▼
      继续循环         返回最终回答
   （最多 N 轮）

关键点: 每一轮 LLM 都可以选择"继续用工具"或"直接回答"
        Agent 不做任何硬编码判断，完全由 LLM 决定
```

**具体例子 — 多步工具调用:**

```
用户: "北京现在几度？如果低于10度帮我算一下穿3件衣服够不够"

第1轮: LLM 选择 → web_search("北京 实时温度")
       工具返回 → "北京当前气温 5°C"

第2轮: LLM 看到 5°C < 10°C，选择 → calculator("5 < 10")
       工具返回 → "True"

第3轮: LLM 综合所有信息，不再调工具
       最终回答 → "北京现在5°C，低于10度。穿3件衣服建议加一件外套。"
```

### 3.2 ReflectionAgent — 自我迭代优化

```
适用场景: 需要高质量输出的生成任务
比如: "写一个高性能的排序函数" "写一篇文章"

运行流程:
              ┌──────────────┐
              │   用户任务    │  "写一个质数判断函数"
              └──────┬───────┘
                     ▼
              ┌──────────────┐
              │  初始生成     │  LLM 写出第一版代码
              │  (Generator)  │
              └──────┬───────┘
                     ▼
         ┌────── 迭代循环 ──────┐
         │                      │
         │   ┌──────────────┐   │
         │   │   反思评审     │   │  另一个 prompt 角色：严格的评审专家
         │   │  (Reflector)  │   │  → "时间复杂度可以优化，用筛法替代试除法"
         │   └──────┬───────┘   │
         │          │           │
         │     "无需改进"?       │
         │      │       │       │
         │     是      否       │
         │      │       │       │
         │      ▼       ▼       │
         │    结束   ┌────────┐ │
         │          │ 优化重写 │ │  根据反馈生成改进版
         │          │(Refiner)│ │
         │          └────┬───┘ │
         │               │      │
         └───────────────┘      │
                                │
                     ┌──────────┘
                     ▼
              返回最终版本

关键点: 不用工具！纯靠 LLM 自我对话
        一个 LLM 扮演两个角色（生成者 + 评审者）反复博弈
        直到评审者说"无需改进"或达到最大轮次
```

**具体例子:**

```
任务: "写一个判断质数的函数"

初始生成: def is_prime(n):
             for i in range(2, n):    ← O(n) 暴力
                 ...

第1轮反思: "时间复杂度 O(n)，应该只遍历到 sqrt(n)"
第1轮优化: for i in range(2, int(n**0.5)+1):  ← O(√n)

第2轮反思: "可以先排除偶数，进一步优化"
第2轮优化: if n % 2 == 0: return False
           for i in range(3, int(n**0.5)+1, 2):

第3轮反思: "无需改进"  → 结束
```

### 3.3 PlanSolveAgent — 先规划后执行

```
适用场景: 复杂的多步推理问题
比如: "对比分析三家公司的财报" "设计一个微服务架构"

运行流程:
              ┌──────────────┐
              │   用户问题    │  "对比苹果和微软的市值"
              └──────┬───────┘
                     ▼
        ┌────────────────────────┐
        │       规划阶段          │  LLM 输出一个步骤列表:
        │      (Planner)         │  ["1.查苹果市值", "2.查微软市值", "3.对比分析"]
        └────────────┬───────────┘
                     ▼
        ┌────────────────────────┐
        │       执行阶段          │  按顺序逐步执行:
        │      (Executor)        │
        │                        │
        │  步骤1 → 调 LLM/工具   │  → "苹果市值: 3.5万亿"
        │  步骤2 → 调 LLM/工具   │  → "微软市值: 3.1万亿"
        │  步骤3 → 调 LLM        │  → "苹果比微软高4000亿..."
        │                        │
        │  每步的结果会传给下一步   │  ← 上下文累积
        └────────────┬───────────┘
                     ▼
              返回最终答案

关键点: 先想清楚要做几步，再一步步执行
        每步执行时能看到前面所有步骤的结果
        适合需要"分而治之"的复杂问题
```

### 三者对比总结

```
                   ReAct             Reflection         PlanSolve
                ─────────────     ─────────────     ─────────────
核心模式         循环决策            自我博弈            分步执行
是否用工具       ✅ 是（核心）       ❌ 否              🔶 可选
LLM 调用次数     不确定(1~N轮)      确定(2×迭代次数)    确定(1+步骤数)
决策者           LLM 每轮自己判断    评审者判断           规划者一次确定
适合场景         需要外部信息        需要高质量输出       复杂多步推理
典型问题         "帮我查/算..."     "帮我写/优化..."    "帮我分析/对比..."
```

## 四、记忆系统如何运转

记忆系统在**多轮对话**中发挥作用。不是一次问答，而是用户和 Agent 持续对话时:

```
对话第1轮
  用户: "我叫小明"
  Agent: "你好小明！"
  → 自动存入短期记忆: "用户叫小明" (importance=1, access_count=0)

对话第2轮
  用户: "帮我搜索新闻"
  Agent 构建上下文时:
    system prompt + [短期记忆: "用户叫小明"] + 历史对话 + 当前输入
  → LLM 能看到用户叫小明，可以个性化回答

... 多轮对话后 ...

对话第10轮
  Agent 调用 memory_cycle():
    1. decay_short()   → 超过30分钟的短期记忆被清理
    2. consolidate()   → importance + access_count >= 5 的短期记忆 → 升级为长期记忆
    3. promote_core()  → importance + access_count >= 10 的长期记忆 → 升级为核心记忆

记忆层级:
  ┌─────────────────────────────────────────────┐
  │ 核心记忆 (永不过期)                          │
  │ "用户是Python开发者，偏好简洁代码风格"        │  ← 反复出现的关键信息
  ├─────────────────────────────────────────────┤
  │ 长期记忆 (持久保留)                          │
  │ "用户正在开发一个电商项目"                    │  ← 重要但不是最核心的
  ├─────────────────────────────────────────────┤
  │ 短期记忆 (30分钟过期)                        │
  │ "用户刚才问了天气" "上一个计算结果是1081"     │  ← 临时上下文
  └─────────────────────────────────────────────┘

构建上下文时的注入顺序:
  [system prompt] → [核心记忆] → [长期记忆] → [短期记忆] → [历史对话] → [用户输入]

越靠前 = LLM 越重视（注意力机制的特性）
所以核心记忆放最前面，短期记忆放后面
```

## 六、为什么用 function calling 而不是正则解析

现在的 reactAgent 用正则解析 LLM 输出:

```
现在的方式（文本解析）:

  LLM 输出:
    "Thought: 我需要计算23*47\n Action: calculator\nAction Input: 23*47"

  Agent 用正则提取:
    re.search(r"Action:\s*(.+)", text)      → "calculator"
    re.search(r"Action Input:\s*(.+)", text) → "23*47"

  问题:
    ❌ LLM 格式稍有偏差就解析失败（多个空格、换行、漏写 Action:）
    ❌ 参数只能是单个字符串，无法传复杂结构
    ❌ 一次只能调一个工具
    ❌ 需要在 system prompt 里写大段格式说明
```

```
改造后的方式（function calling）:

  LLM 返回结构化 JSON:
    {
      "tool_calls": [{
        "function": {
          "name": "calculator",
          "arguments": "{\"expression\": \"23*47\"}"
        }
      }]
    }

  Agent 直接解析 JSON:
    name = tool_call.function.name
    args = json.loads(tool_call.function.arguments)

  优势:
    ✅ LLM API 原生支持，不会格式出错
    ✅ 参数是结构化的 JSON，支持多参数、嵌套结构
    ✅ 一次可以返回多个 tool_calls（并行调用多个工具）
    ✅ 不需要在 prompt 里教 LLM 输出格式
```

## 七、完成后你能做什么

### 开箱即用

```python
# 3行代码启动一个能用工具的 AI 助手
agent = ReActAgent(name="助手", llm=LLMClient(), tools=tools)
answer = agent.run("帮我算一下 sin(30°) 的值")
```

### 快速扩展工具

```python
# 写一个类，注册一行，Agent 自动获得新能力
class EmailTool(Tool): ...
tools.register(EmailTool())
# 现在 Agent 可以发邮件了
```

### 快速扩展 Agent 策略

```python
# 继承 Agent，实现 run()，就是一个新策略
class DebateAgent(Agent):
    """正方 vs 反方辩论，最终综合结论"""
    def run(self, topic: str) -> str: ...
```

### 切换 LLM 后端

```python
# 换个环境变量，同一套 Agent 跑不同模型
llm = LLMClient()                                    # 自动检测 .env 配置
llm = LLMClient(base_url="http://localhost:11434/v1") # Ollama 本地
llm = LLMClient(model="gpt-4")                       # OpenAI
llm = LLMClient(model="glm-4")                       # 智谱
# Agent 代码一行不改
```
